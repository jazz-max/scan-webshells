# scan-webshells.py — сканер и лечилка вебшеллов

Поиск, карантин и лечение веб-шеллов/PHP-загрузчиков после взлома сервера.
Обходит дерево **сам** (не зависит от `xargs` и длины списка аргументов),
сканирует **все файлы по содержимому** (не только по расширению),
работает **на всех ядрах**, **печатает находки сразу** (потоково) и показывает прогресс.

Вывод **двуязычный**: по умолчанию язык берётся из `$LANG`, принудительно — флагом `--lang en|ru`.

---

## Быстрый старт

```bash
# 1. Посмотреть, что заражено (ничего не меняет, занимает все ядра, имена идут сразу)
python3 scan-webshells.py /var/www | tee found.txt

# 2. ПРЕДПРОСМОТР лечения + карантина — увидеть план, не трогая файлы  ← делать первым
python3 scan-webshells.py /var/www --quarantine /root/Q --clean-index --dry-run | tee plan.txt

# 3. БОЕВОЙ запуск: вылечить index.php + унести цельные шеллы в карантин
python3 scan-webshells.py /var/www --quarantine /root/Q --clean-index | tee clean.log
```

Требуется только Python 3 (стандартная библиотека, без внешних пакетов).

## Флаги

| Флаг | Что делает |
|---|---|
| `path` | каталог или файл для сканирования (по умолчанию текущий) |
| `--quarantine DIR` | перенести **цельные шеллы** в `DIR` (с сохранением путей) |
| `--clean-index` | вырезать инъекцию из заражённых `index.php` (бэкап в `.bak`) |
| `--dry-run` | только показать, что было бы сделано — **ничего не менять** |
| `--quiet` | без индикатора прогресса (для cron/логов) |
| `-j N`, `--jobs N` | число параллельных процессов (по умолчанию = число ядер) |
| `--lang en\|ru` | язык вывода (по умолчанию — авто из `$LANG`) |
| `-h`, `--help` | справка |

## Вывод и прогресс

- **Находки печатаются в реальном времени** (в `stdout`), как только обнаружены:
  ```
  === НАЙДЕНО (выводится по мере обнаружения) ===
  [INJECT] /var/www/.../index.php   =>  goto-загрузчик A
  [FULL]   /var/www/.../accesson.php =>  accesson бэкдор
  ```
- **Прогресс** идёт в `stderr` отдельной перезаписываемой строкой и показывает процент.
- Прогресс **автоматически отключается**, если вывод не в живой терминал (`| tee`, `> файл`, `| grep`). Принудительно — флагом `--quiet`.
- **Ctrl+C** останавливает корректно: печатает «ПРЕРВАНО … найдено N» без traceback; при прерывании действия (`--quarantine`/`--clean-index`) **не выполняются**.

## Рекомендуемый порядок на проде

```bash
# 1. посмотреть план (записать в файл — будет полный список даже если терминал закроется)
python3 scan-webshells.py /var/www --quarantine /root/Q --clean-index --dry-run | tee /root/plan.txt
# 2. выполнить (дать доработать до конца, НЕ прерывать)
python3 scan-webshells.py /var/www --quarantine /root/Q --clean-index | tee /root/clean.log
# 3. проверить, что сайты работают
# 4. только потом удалить бэкапы и карантин
find /var/www -name '*.bak' -newermt '-1 hour'
```

## Как классифицируются находки: FULL vs INJECT

- **`[FULL]`** — файл **целиком вирус** (`accesson.php`, `cache.php`, `.tmp`, медиа-пустышки, BOM-вебшеллы). Их забирает `--quarantine` (перенос = бэкап).
- **`[INJECT]`** — **легитимный файл сайта с вставкой сверху** (например главный `index.php`, `vendor/composer/autoload_real.php`). Его `--quarantine` **не трогает**, а `--clean-index` вырезает только вредоносный `<?php…?>`-блок в начале и кладёт оригинал в `.bak` рядом.

Множества не пересекаются — `--clean-index` и `--quarantine` можно запускать вместе.

## Forensic: даты файлов и манифест

`--quarantine` сохраняет **mtime** (когда файл записан/появился) при переносе и пишет CSV-манифест `_quarantine_manifest.csv`:

```
original_path,mtime,ctime,size,signatures
/var/www/.../accesson.php,2026-05-12 03:41:09,2026-05-12 03:41:09,212,accesson backdoor
```

- **mtime** — основной признак «когда появился файл», переносится без изменений;
- **ctime** при любом переносе меняется — поэтому снимается **до** переноса;
- mtime атакующий мог подделать (`touch`) — сверяйте с логами веб-сервера.

## Что детектируется

- goto-обфусцированные загрузчики (range-инъекция, скрытые метки, hex/octal-строки);
- `Sy1Lz`-упакованные шеллы; WSO/FilesMan; xleet; `accesson.php`; AES файл-менеджеры;
- base64/`zip://`-инклюды медиа-пустышек; WP-маскировка `eval(pre_term_name())`;
- BOM-вебшеллы; обфускация `/*-…-*/`; hex-кодированные суперглобалы;
- PHP в медиа/`.txt` (только при опасных функциях); `.zip` с `.tmp` внутри;
- Joomla-кампания: переинфектор `$zym_decrypt` в `vendor/composer/autoload_real.php` + дроп файл-менеджеров в папках-фамилиях (`zhou/zheng/wu/...`) с `.htaccess`-реактиватором; ROT13- и китайские шеллы, `EVAL()`-обфускация, непроверенные Uploadify-загрузчики, ввод напрямую в `eval/system`.

> **Composer-автозагрузчики проверяются даже внутри `vendor/`** — это типичная цель переинфекторов. Остальной `vendor/`, `node_modules/`, `.git/` пропускаются для скорости. Из каждого файла читаются первые 3 МБ.

Сканирование идёт по **содержимому** — шелл под именем `logo.png` тоже найдётся.

## Ложные срабатывания

- WSO `$▛`-переменные и `/*-`-обфускация — **только в PHP-файлах** (иначе ловят JPEG/бинари/JS-библиотеки);
- «PHP в медиа/txt» — только если `<?php` в начале файла (первые 200 байт) **и** есть опасная функция (отсекает codemirror/HTMLPurifier);
- легитимный установщик Akeeba — в белом списке.

Если что-то помечено ошибочно — видно по типу сигнатуры; правьте `SIG`/`WHITELIST` в начале скрипта.

---

## Важные предупреждения

- Карантин-папку (`/root/Q`) держите **вне** сканируемого дерева.
- `.bak` и карантин удаляйте **после** проверки работоспособности сайтов.
- Файлы могут быть `chmod 444` (приём малвари) — скрипт снимает read-only сам; если владелец не вы (`www-data`) — запускайте через `sudo`.
- Это **RCE-компрометация**: смените пароли/ключи БД и API, проверьте `crontab`, `uploads/`, сверьте код с git/бэкапом.

---

## Ограничения

- Сигнатурный детектор настроен на конкретный набор семейств; для иной малвари добавляйте сигнатуры в `SIG`.
- Действия (`--quarantine`, `--clean-index`) однопоточные; параллелится только сканирование.
